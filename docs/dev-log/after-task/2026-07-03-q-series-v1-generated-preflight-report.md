# After Task: Q-Series v1 generated preflight report

## 1. Goal

Make the Q-Series v1.0 release preflight reviewable from a generated file, not
only from command output or chat, while preserving the current no-claim
boundary.

## 2. Implemented

Extended `tools/qseries_v1_release_check.py` with `--write-report` and
`--check-report`. The generated report is
`docs/dev-log/release-audits/q-series-v1-preflight-report.md`. It records the
full preflight result, the row-accounting percentages, the routine command, and
the boundary that prevents the v1.0 practical surface from becoming coverage,
`supported`, REML, AI-REML, q4/q8, broad bridge-support, or public-support
wording.

Updated the focused `structured-re-conversion-contracts` test to read the
generated report and check the ledger, claim guard, Mission Control,
row-accounting, and no-claim boundary phrases. Updated the dashboard README so
routine Q-Series v1.0 preflight uses `--check-report`, and `--write-report` is
reserved for intentional release-boundary updates.

## 3a. Decisions and Rejected Alternatives

The report is generated by the existing consolidated preflight instead of a
separate renderer. That keeps one command responsible for the release-prep
evidence packet while leaving the ledger generator, claim guard, and Mission
Control validator independent.

I did not make the focused R test regenerate the full report because that would
launch Mission Control recursively inside the contract. The R test reads the
full generated report and separately runs the non-recursive preflight mode.

## 9. What Did Not Go Smoothly

The main subtlety was separating the full generated report, which should record
Mission Control `ok`, from the non-recursive preflight mode used in the focused
R contract. The file is generated by full preflight; the nested test reads it
without rewriting it.

## 10. Known Residuals

This is still Q-Series release-prep evidence, not full drmTMB v1.0 release
proof. Final v1.0 readiness still needs package-level checks and reviewer
approval for public release wording.

## 11. Team Learning

For repeated release gates, keep both a one-command preflight and a generated
report. The command catches drift; the report gives reviewers a stable artifact
to inspect.
