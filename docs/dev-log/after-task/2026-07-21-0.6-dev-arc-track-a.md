The `||` defect was not cosmetic. R parses `||` as its own operator, so
`is_random_bar_call()` (matching only `as.name("|")`) never saw it and
`flatten_plus_terms()` (recursing only through `+`) passed it through whole. The
term reached the **fixed-effect** design matrix and aborted with
`'length = N' in coercion to 'logical(1)'`, whose N is the row count — so it read
as a size problem and was not. It also escaped `drm_reject_phase1_terms()`,
because `formula_contains_call(rhs, "|")` compares the call name `"|"` and a `||`
call is named `"||"`.

## 3a. Decisions and Rejected Alternatives

**D-A1: desugar at the `drmTMB()` entry, not in the splitters.** The natural site
looked like `extract_random_mu_terms()` / `extract_random_sigma_terms()`. Rejected:
both take only `(rhs, dpar)`, so `data` is out of scope and the numeric-vs-factor
slope test is impossible there; and there are ~40 call sites versus one. The entry
hook also serves the Julia bridge, which returns before the family routes.

**D-A2: reject categorical slopes under `||` rather than desugar them.** In lme4
`||` splits by formula **term**, not by design-matrix **column**, so a factor slope
keeps its within-factor correlations and is not in fact uncorrelated. Silently
copying that wart was rejected in favour of an error naming the explicit form.

**D-A3: `-1` errors rather than being interpreted.** Loud beats clever; a
last-wins reading of `-1` would be a guess about intent.

**D-A4: A3 dropped as scoped.** Decision D3 held that the missing-data slice was
"WIRED + TESTED but unclaimed — claim the free win." Rose returned **PREMISE
FALSE**: it is claimed on six surfaces, generated from the ledger, and calibrated
at or below its evidence. A3's own acceptance test (ledger 37/37, includes
regenerated not hand-edited, claim ≤ cited tests) **already passed on an empty
diff**. `cells.tsv` and the generated includes were not touched. Rejected
alternative: doing the surfacing anyway "for completeness" — that would have
manufactured a diff to justify a staffed slice.

**D-A5: the MCAR/MAR omission is referred to Fisher, not fixed here.** MCAR/MAR is
never named in `vignettes/`, `R/`, or `man/`, though every `mr-*` `claim_boundary`
rests on fixed-seed 25% MCAR masking. Rose classifies this as an omission running
the **safe** direction, not an over-claim. Whether a methods package may ship a
FIML missing-data article that never states its missingness mechanism is an
inference call. Rejected: quietly adding an assumption sentence.

**D-A6: the stale `phase18-actions-runner` tests are filed, not fixed.** Eleven
test blocks assert content D-50 deliberately removed. Rewriting them inside Track A
was rejected as scope creep in someone else's file; the right fix converts them
into a guard that the workflow *stays* disabled, which is its own change.

## 9. What Did Not Go Smoothly

**Two self-inflicted regressions, both caught, neither by me first.**

1. Rewording the Gaussian dpar message to "support only …" broke
   `test-gaussian-location-scale.R:403`, which matches the loose substring
   `"only support"` — a substring **12 assertions across 9 test files** depend on.
   My pre-flight grep searched for the full cli string and for "Phase 1 Gaussian",
   so it could not have found this. Fixed by changing only the two words A2
   actually required.
2. `expand_double_bar_term()` decided the intercept with the order-insensitive
   `!any(is_zero)`. R formula algebra is **last-wins**, so `(0 + 1 + x || g)`
   silently dropped a variance component — logLik −436.8 df 4 against lme4's
   −298.9 df 5, discarding an intercept SD of 0.843, with no warning. Worse, the
   single-bar sibling `(0 + 1 + x | g)` is loudly *rejected*, so the desugaring was
   strictly **more permissive than the route it desugars onto**. Found by an
   adversarial review, confirmed by an independent verifier against
   `lme4:::expandDoubleVerts`, and fixed to last-wins semantics.

**testthat's failure cap masked coverage twice.** The first full run reported exit
code 0 while the log ended "Maximum number of 10 failures reached, some test
results may be missing." I reported progress from the exit code before reading the
log — the same class of error this arc exists to correct. Neither
`options(testthat.progress.max_fails=)` nor `TESTTHAT_MAX_FAILS` lifted the cap;
the working route was `test_local(filter=…, invert=TRUE)` to exclude the file
consuming the budget.

**The permission classifier was down or intermittent for most of the session**,

## 11. Team Learning

**The false-positive pattern is now five for five, and it has one shape: a brain
note asserted a repo state, and the repo was not checked before that assertion
became a locked decision.** D3's source note was already stale against an
after-task report sitting in the same repo. Rose's proposed mechanical safeguard:
*before locking any decision of the form "X is unclaimed", run its acceptance test
first.* Here that was three commands and 0.3 seconds and would have deleted the
slice before it was staffed. It belongs in Phase 0.25 as a required column —
"decision falsifier run, result."

**Message-substring coupling is a live hazard in this repo.** Tests match short
fragments of cli messages, so grepping for the full string proves nothing. Any
message edit needs a substring sweep across `tests/`, `vignettes/`, and `man/`.

**Adversarial verification earns its cost.** Five review lenses raised 11
findings; independent refutation reduced them to 1. That 1 was a real silent-
wrongness bug in new code that all my own targeted tests had passed. Without the
refutation step the other 10 would have generated busywork.

**Verify with the ecosystem's own oracle where one exists.** Once `||` was sold as
lme4 compatibility, `lme4:::expandDoubleVerts()` became the correct arbiter — not
my reading of what the spelling ought to mean.

## 12. Cross-Product Coverage

`||` desugaring and smooth rejection are **cross-cutting formula transformations**:
they run on every `drmTMB()` call, for every family, dpar, engine, and provider.
That is the Section-12 hazard class (the REML arc), so the negative space matters
more than the positive result.

**What is covered:** univariate Gaussian `mu`, numeric slopes, single and multiple
slopes, implicit/explicit/absent intercepts across all nine 0/1 orderings, factor
slope rejection, non-symbol slope rejection, compound-group rejection, and the mu
and sigma grouping guards. Equivalence measured on one 20-group Gaussian fixture.

**This slice does NOT cover:**

- **Families.** Only Gaussian was fit with `||`. The other 17 response families are
  untested with this spelling, though the rewrite runs for all of them.
- **Distributional parameters.** No `||` fit on `sigma`, `nu`, `zi`, `hu`, `zoi`,
  `coi`, `shape`, `skew`, or `rho12`.
- **Bivariate.** No `||` fit on `mu1`/`mu2`, and no interaction tested with
  `rho12` or labelled covariance blocks.
- **Structured providers.** No `||` inside or alongside `phylo()`, `spatial()`,
  `animal()`, `relmat()`, or `phylo_interaction()`.
- **Engines.** The Julia bridge receives the rewritten formula but was never
  exercised with `||`.
- **REML.** No `||` fit with `REML = TRUE`, where random/fixed partitioning is
  exactly what the rewrite alters.
- **Missing data.** No `||` fit under `miss_control()`, and `impute_model()`
  formulas bypass the hook entirely (see 10).
- **Weights, offsets, `meta_V()`, `corpair()`, `sd()` formulas** in combination
  with `||`.
- **Estimator correctness.** Equivalence is to drmTMB's own single-bar route and to
  lme4's rewrite; no coverage, interval, or recovery claim is made or implied.

Nothing above is asserted to work or to fail — it is untested, and the honest
reading is that `||` is verified on the Gaussian `mu` route only.
