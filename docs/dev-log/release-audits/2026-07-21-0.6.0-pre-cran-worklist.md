## Rung 1b — first-impression surface (finite; cheap now, costly after a first release)

An earlier draft of this worklist split the work in two — release integrity (in scope) versus capability
breadth (post-0.6). That binary was wrong, and it missed the category below. These items are neither
packaging nor new science: they are the **API and error surface a newcomer meets first**. They are cheap
to fix before a first CRAN release and expensive afterwards, because once the package is on CRAN the
grammar is a compatibility promise and deprecation cycles are owed.

- [ ] **`||` fails with a raw internal error.** `bf(y ~ x + (1 + x || id))` — the uncorrelated
      random-slope spelling an lme4/brms/glmmTMB user types first — aborts with
      `'length = 320' in coercion to 'logical(1)'`. Verified 2026-07-21 on a toy Gaussian fit. It is
      **not** a silent wrong answer (the correlated `|` and two-term `(1 | id) + (0 + x | id)` forms both
      behave correctly), but the message names neither the unsupported operator nor the working
      alternative. Either add a clean `cli::cli_abort` pointing to the two-term form, or adopt `||` as
      first-class per **#776**. *Check:* the `||` formula produces an actionable message naming the
      supported spelling, or fits with the documented uncorrelated semantics.
- [ ] **Newcomer-syntax sweep.** The `||` defect was found in ~30 seconds by typing what a glmmTMB user
      would type. Sweep the rest: the formulas a reader arrives with from `lme4`, `glmmTMB`, `brms`, and
      `gamlss`, and confirm each either fits or fails with a message naming the supported alternative.
      Repo doctrine already requires this ("planned-feature errors should appear before fitting"; explain
      what users should try next when syntax is unsupported). *Check:* a recorded table of attempted
      formulas → outcome, with no raw internal R errors surviving.
- [ ] **#806 — silent `NULL` from documented extractors.** `df.residual()` returns `integer(0)`,
      `vcov()`/`fitted()` return `NULL` on `new_drmTMB_julia_xfam()` objects. Silent wrong-typed output is
      worse than an error. Either wire them correctly or make them error explicitly. *Check:* no exported
      method returns a silent `NULL`/length-zero for an object it accepts. **Maintainer decision.**
- [ ] **`coef.drmTMB()` discoverability.** The dpar-keyed list return is internally coherent and tested,
      but has no direct Rd alias or example. Do not change the return type during the release lane; add
      documentation. *Check:* `?coef.drmTMB` shows a runnable example of the list structure.

## Scope fence — what is NOT on this list (post-0.6)

These are real and worthwhile, but they are capability *breadth*, and their tail is unbounded (each needs
a DGP, a compute campaign, and a D-43 panel). Pulling any into the release turns a finite job open-ended.
They belong to 0.7+, per the manifest's DEFER table:

- Coverage-certifying the row-specific regression-`rho12` intervals (#802).
- Promoting further ledger cells.
- Mixed-family bivariate models and `engine = "julia"` parity (#499).
- Any new family, structured slope, or interval method.
- The gllvmTMB q2 phylogenetic binomial-logit probe (a post-0.6 design lead only; it establishes no
  drmTMB capability).

0.6.0's value proposition is an honest, deliberately narrow surface. Breadth is a 0.7 conversation —
but a **confusing first error message is not breadth**, it is rung 1b, and it belongs before CRAN.
